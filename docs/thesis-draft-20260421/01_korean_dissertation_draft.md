## 1. 국문초록

프로그래밍 교육에서 블록 기반 환경은 초보 학습자의 진입 장벽을 낮추는 데 효과적이지만, 실제 텍스트 언어로의 전환을 충분히 지원하지 못한다는 한계가 있다. 반대로 Python과 같은 텍스트 기반 범용 언어는 확장성과 실용성이 높지만, 설치와 환경 구성, 패키지 의존성, 초기 문법 오류, 디버깅 부담으로 인해 학습 초기 단계에서 높은 인지적 비용을 유발한다. 본 연구는 이러한 간극을 해소하기 위해 양방향 블록-텍스트 변환, 브라우저 내 Python 인터프리터, JavaScript 기반 실행 브리지, AI 기반 라이브러리 확장, 학습 로그 분석을 통합한 교육용 다중표현 프로그래밍 시스템을 설계하고 평가한다. 제안 시스템은 Blockly 기반 시각 편집기와 Python 편집기가 동일한 의미 구조를 공유하도록 설계하였으며, 블록에서 생성된 Python 코드와 학습자가 직접 수정한 Python 코드가 하나의 실행 파이프라인으로 수렴되도록 구성하였다. 또한 WebAssembly 기반 Python 런타임과 브라우저 이벤트 브리지를 결합하여 서버 의존성을 줄이면서도 설치 없는 실행과 상호작용적 프로그래밍을 지원한다. 본 연구는 시스템 차원에서 실행 충실도, 반응성, 패키지 지원 범위, 확장 가능성을 평가하고, 교육 차원에서 학습 성과, 블록-텍스트 전환 능력, 자기효능감, 인지부하, 사용성, 코드 품질을 검증한다. 본 연구의 의의는 블록 기반 입문 환경과 범용 Python 실행 환경 사이의 의미적 단절을 완화하고, `입문-전환-확장`을 하나의 학습 연속선 위에서 다룰 수 있는 프로그래밍 언어 시스템을 제안했다는 데 있다.

## 2. 핵심어

프로그래밍 교육, 블록 기반 프로그래밍, Python, WebAssembly, Pyodide, Blockly, 양방향 변환, 학습 분석, 한국어 친화 프로그래밍, 교육용 실행환경

## 3. 1장 서론

프로그래밍 교육은 더 이상 일부 전공자의 전유물이 아니다. 초중등 교육에서의 컴퓨팅 소양 강화, 대학에서의 데이터·AI 기반 문제해결 역량 요구, 산업 현장에서의 소프트웨어 활용 확산은 프로그래밍을 보편적 기초 역량으로 재배치하고 있다. 그러나 프로그래밍을 누구나 배워야 한다는 사회적 합의와, 실제로 누구나 쉽게 배울 수 있는지는 별개의 문제다. 초보 학습자는 개념 자체의 난이도뿐 아니라 설치, 실행환경 설정, 패키지 관리, 경로 문제, IDE 사용법, 문법 오류 해석 등 언어 외적 요소로 인해 학습 초기에 큰 좌절을 경험한다. 특히 Python은 교육 현장에서 널리 사용되는 언어이지만, 환경 구축의 번거로움과 텍스트 문법 중심의 학습 곡선은 여전히 초심자에게 높은 장벽이 된다.

이 문제를 완화하기 위해 블록 기반 프로그래밍 환경이 널리 사용되어 왔다. Scratch와 Blockly 계열 환경은 문법 오류를 구조적으로 차단하고, 프로그램의 제어 흐름과 데이터 흐름을 시각적으로 드러내며, 즉각적인 피드백을 제공함으로써 학습 초기의 인지적 부담을 낮춘다. Blockly 공식 문서는 블록을 직접 실행하는 것이 아니라 블록을 코드 문자열로 생성해 실행한다고 설명하며, JavaScript와 Python을 포함한 여러 언어용 코드 생성기를 제공한다 [R1]. 이 구조는 블록 환경이 단지 시각적 장난감이 아니라, 적절한 코드 생성기와 실행기를 결합하면 실제 프로그래밍 학습 도구가 될 수 있음을 시사한다.

그러나 블록 기반 환경의 성공이 곧바로 텍스트 기반 프로그래밍 역량의 형성으로 이어지는 것은 아니다. 최근 체계적 문헌고찰은 블록에서 텍스트로의 전환 지원이 독립된 연구 주제로 성장했음을 보여 주며, 표현 간 연결을 드러내는 기술적 장치와 과제의 의미성, 그리고 수업 차원의 지원이 함께 필요하다고 정리한다 [R2]. 같은 문제의식 아래 수행된 선행연구들은 하이브리드 또는 구조화 편집기가 전환 과정에서 문법 오류와 자료형 관련 오류를 줄일 수 있음을 보고했지만 [R3][R4], 많은 시스템은 여전히 한쪽 표현을 보조적으로만 제공하거나, 전환 후의 실행 의미를 일관되게 유지하지 못한다. 즉, 학습자는 블록에서는 동작하던 프로그램이 텍스트로 넘어가는 순간 전혀 다른 환경과 규칙을 만나게 되며, 이때 전이 비용이 급격히 커진다.

텍스트 환경을 브라우저 내부로 가져오려는 시도 역시 중요한 흐름이다. Stanford의 PyodideU는 Python IDE와 런타임을 데스크톱, 노트북, 태블릿, 모바일 브라우저에서 서버 없이 실행할 수 있음을 보여 주었고, `WebAssembly blocking problem`을 해결해 상호작용적 Python 실행을 제공했다고 보고한다 [R5]. Pyodide 공식 문서도 Web Worker 내부에서 Python 스크립트를 비동기로 실행하는 예제를 제공하며, 메인 스레드에서 실행할 경우 UI가 막힐 수 있으므로 Worker 사용이 유리하다고 설명한다 [R6]. 또한 Pyodide는 JavaScript 객체를 Python 모듈로 등록할 수 있는 `registerJsModule` API를 제공하고 [R7], 내장 패키지와 `micropip` 기반 패키지 로딩을 지원한다 [R8]. 이러한 발전은 브라우저 내부에서 실제 Python 실행환경을 제공하는 것이 더 이상 아이디어 수준이 아니라 구현 가능한 선택지임을 보여 준다.

그럼에도 현재의 교육용 시스템들은 몇 가지 공백을 남긴다. 첫째, 블록 표현과 Python 표현이 모두 존재하더라도, 두 표현이 동일한 의미 구조를 공유하도록 설계된 사례는 제한적이다. 일방향 코드 내보내기는 흔하지만, 학습자가 수정한 Python 코드를 다시 블록 구조로 복원하고 학습 흐름 속에서 재사용하는 양방향성은 기술적으로 어렵고 연구적으로도 충분히 축적되지 않았다. 둘째, 브라우저 내부 Python 런타임을 제공하는 시스템은 존재하지만, 블록 기반 입문 환경과 범용 Python 실행환경, 이벤트 기반 시각 실행, 라이브러리 확장, 학습 로그 분석을 하나의 플랫폼에서 통합한 사례는 드물다. 셋째, 비영어권 학습자의 입장에서는 언어적 장벽이 여전히 크다. 오류 메시지, 라이브러리 문서, 코드 예시, 인터페이스 표현이 영어 중심인 상황에서, 한국어 친화적 도구화는 단순 번역 문제가 아니라 언어 설계 및 교육 접근성의 문제다.

부산대학교 우균 교수의 공개 연구 이력은 이러한 문제를 단순한 교육서비스가 아니라 프로그래밍 언어 시스템의 문제로 다루는 방향을 뒷받침한다. 공식 프로필은 전공분야를 `프로그래밍 언어 및 컴파일러`로 제시하며, 연구실 소개는 명령형, 함수형, 논리, 객체지향 언어와 프로그램 분석을 연구범위로 설명한다 [R9][R10]. 또한 최근 공개된 졸업과제 안내에서는 한글 함수형 언어 `머꼬`, `KoBASIC`, `i새싹` 인터프리터, Python 모듈 확장, GUI 설치관리자 등을 제안하고 있다 [R11]. DBLP에 수록된 대표 논문 역시 소스코드 표절 탐지, 소스코드 관리 시스템, 컴파일러 테스트 축소, 코드 스타일 시각화 등 `표현을 내부 구조로 바꾸고, 분석 가능하게 만들고, 실행과 검증을 연결하는` 흐름을 보여 준다 [R12]. 따라서 본 연구는 블록 기반 코딩 교육 도구를 만드는 작업이 아니라, 교육 맥락에서 작동하는 프로그래밍 언어·실행환경·변환기의 통합 설계라는 관점에서 접근할 필요가 있다.

본 연구는 이러한 배경에서 `브라우저 내 Python 인터프리터`, `양방향 블록-텍스트 변환`, `JavaScript 브리지 기반 상호작용 실행`, `AI 기반 라이브러리 블록 생성`, `학습 로그 분석`을 통합한 교육용 다중표현 프로그래밍 시스템을 제안한다. 제안 시스템의 핵심 원칙은 표현 양식이 달라도 실행 의미는 일관되어야 한다는 점이다. 학습자는 블록을 조합하여 프로그램을 만들 수도 있고 Python 텍스트를 직접 편집할 수도 있다. 시스템은 두 표현을 상호 변환 가능한 상태로 유지하며, 최종적으로는 동일한 Python 실행 파이프라인 위에서 동작하게 한다. 이 구조를 통해 블록은 입문용, 텍스트는 고급용이라는 단선적 구분을 넘어서, 학습자가 필요에 따라 표현을 왕복하며 개념과 실행을 함께 이해하도록 지원한다.

제안 시스템은 다음과 같은 설계 목표를 갖는다. 첫째, 블록과 텍스트 사이의 전환 비용을 줄이기 위해 의미 일관적 양방향 변환을 제공한다. 둘째, 설치 없는 실행을 위해 브라우저 내부 Python 런타임을 사용하되, 장시간 실행으로 인한 UI 정지를 피하기 위해 Worker 기반 구조를 사용한다. 셋째, 시각적 stage, 입력, 출력, 타이머, 이벤트와 같은 학습 친화적 상호작용 요소를 JavaScript 브리지로 연결한다. 넷째, 외부 라이브러리를 단순 코드 조각 수준이 아니라 `블록 정의 + 코드 생성기 + 런타임 연결정보`를 포함하는 패키지 단위로 다룬다. 다섯째, 학습 로그를 세밀하게 수집하여 블록-텍스트 전환 과정, 디버깅 패턴, AI 의존도, 코드 품질 변화를 실증적으로 분석한다.

이 논문은 다음 네 가지 연구질문을 다룬다.

1. 양방향 블록-텍스트 변환을 갖춘 다중표현 환경은 blocks-only 또는 text-only 환경에 비해 학습자의 Python 전환 성과를 향상시키는가?
2. 브라우저 내부 Python 인터프리터와 이벤트 브리지를 결합한 실행 모델은 교육용 상호작용 과제에서 충분한 실행 충실도와 반응성을 제공하는가?
3. AI 기반 라이브러리 확장 및 설명 보조 기능은 학습 효율을 높이되, 과도한 의존 없이 생산적인 사용 패턴을 유도하는가?
4. 블록-텍스트 전환 로그, 오류 유형, 실행 패턴은 학습 성과와 어떠한 관계를 가지는가?

본 연구의 공헌은 다음과 같다. 첫째, 블록 기반 입문 환경과 텍스트 기반 Python 실행 환경 사이의 의미적 단절을 완화하는 양방향 다중표현 프로그래밍 아키텍처를 제안한다. 둘째, WebAssembly 기반 Python 런타임과 JavaScript 이벤트 브리지를 결합한 브라우저 상주 실행환경을 구현함으로써 서버 의존성을 낮추고 상호작용적 실행을 제공한다. 셋째, 라이브러리 기능을 블록 패키지와 실행 브리지 수준에서 확장 가능한 형태로 모델링하고, AI를 이를 위한 생성·보조 도구로 통합한다. 넷째, 전통적인 정답률 중심 평가를 넘어 전환 능력, 유지 효과, 로그 기반 행태, 자기효능감, 인지부하, 사용성을 포함하는 다층 평가 프레임을 제시한다.

결과적으로 본 연구는 블록 기반 프로그래밍과 Python 기반 범용 프로그래밍을 대립시키지 않고, 하나의 실행 의미 위에 여러 표현을 포개는 교육용 프로그래밍 시스템을 목표로 한다. 이 접근은 교육공학적 유용성뿐 아니라, 프로그래밍 언어 설계, 코드 생성, 실행환경 통합, 사용자 상호작용 브리지, 패키지 관리, 로그 기반 분석이라는 컴퓨터과학적 문제들을 함께 다룬다는 점에서 학술적 의의를 갖는다.

## 4. 2장 관련연구

### 4.1 블록 기반 환경과 블록-텍스트 전환

Blockly는 블록을 직접 실행하지 않고, 블록을 코드 문자열로 생성해 실행하는 구조를 채택한다 [R1]. 이는 블록 환경이 본질적으로 코드 생성기와 실행기의 결합체임을 의미한다. 최근 체계적 문헌고찰은 블록에서 텍스트로의 전환 연구를 분석하며, 표현 간 연결을 드러내는 기술적 지원, 과제의 의미성, 학습 동기, 교사의 개입, 그리고 더 강한 실험적 증거가 필요하다고 정리한다 [R2]. 하이브리드 환경 비교 연구는 전환 용이성과 기초 개념 이해 측면에서 hybrid 기반 접근이 blocks-only 대비 우세할 수 있음을 보고한다 [R3]. 구조화 편집기 연구는 Scratch에서 Python으로 전환하는 과정에서 구조화된 전이 도구가 구문 오류와 자료형 오류를 상당히 줄일 수 있음을 보여 준다 [R4]. 그러나 다수 연구는 전환 시점의 편집 지원에 초점을 맞추고 있으며, 블록과 텍스트가 하나의 실행 의미를 공유하는지까지는 충분히 다루지 않는다.

### 4.2 브라우저 내부 Python 실행

PyodideU는 교육용 Python IDE와 런타임을 브라우저 내부에서 동작시키며, 서버 기반 런타임이 필요 없는 Python 교육 환경의 실현 가능성을 보여 준다 [R5]. Pyodide 공식 문서는 Worker에서 Python 코드를 실행하는 방식을 제시하고, 메인 스레드 차단을 피하기 위한 아키텍처를 권장한다 [R6]. 또한 `registerJsModule`을 통해 JavaScript 객체를 Python 모듈처럼 import할 수 있으며 [R7], `loadPackage`, `loadPackagesFromImports`, `micropip.install`을 통해 패키지 로딩을 수행한다 [R7][R8]. 이러한 특성은 교육용 stage, sprite, 입력 이벤트, 파일 시스템, 출력창 등을 Python 코드와 연결하는 데 유리하다. 다만 브라우저 런타임은 운영체제 자원 접근, GUI, 장치 I/O, 복잡한 네이티브 확장 사용에서 제한이 존재하므로, 교육용 과제 범위에 맞는 브리지 설계가 필수적이다.

### 4.3 한국어 친화형 교육용 프로그래밍 언어

한글 기반 교육용 언어 연구는 프로그래밍 진입 장벽을 단순한 난이도 문제가 아니라 언어적 접근성 문제로 본다. 부산대학교 프로그래밍언어연구실 공개 자료와 최근 과제 안내는 `새싹`, `KoBASIC`, `머꼬`와 같은 한글 프로그래밍 언어, 인터프리터, Python 모듈 확장 등을 지속적으로 다루고 있음을 보여 준다 [R10][R11]. 이는 한국어 학습자를 위한 프로그래밍 환경이 UI 한글화만으로 해결되지 않으며, 언어 설계와 실행환경이 함께 고려되어야 함을 시사한다. 본 연구는 범용 Python 자체를 한글로 바꾸지는 않지만, 학습 인터페이스와 블록 표현, 오류 설명, 라이브러리 가이드, 피드백 경로를 한국어 친화적으로 설계함으로써 이 흐름을 계승한다.

### 4.4 AI 기반 프로그래밍 학습 지원

생성형 AI는 프로그래밍 교육에서 코드 제안, 오류 설명, 힌트 제공, 예제 생성, 자동 피드백의 가능성을 열었지만, 동시에 과의존과 오류 전파, 기초 개념 약화 위험도 함께 가져왔다. 본 연구에서 AI의 역할은 전체 프로그램 자동생성보다 제한적으로 정의한다. 즉, 라이브러리 블록 패키지 생성 보조, API 사용 예시 생성, 설명형 피드백 제공, 블록-텍스트 간 대응 설명, 교사용 실험 자료 준비에 초점을 둔다. 이는 AI를 학습자 대신 코드를 써 주는 대체물로 두기보다, 확장성과 설명성을 높이는 개발·학습 보조 계층으로 배치하기 위함이다.

## 5. 3장 시스템 설계

### 5.1 설계 목표

제안 시스템은 다음 다섯 목표를 갖는다.

1. 블록과 Python 텍스트 사이의 의미 일관성 유지
2. 브라우저 내부 Python 실행을 통한 설치 비용 제거
3. 시각적 상호작용을 위한 이벤트 브리지 제공
4. 라이브러리 패키지의 동적 확장 지원
5. 학습 로그와 행태 분석의 체계적 수집

### 5.2 전체 아키텍처

시스템은 `표현 계층`, `변환 계층`, `실행 계층`, `확장 계층`, `분석 계층`으로 구성된다.

- 표현 계층은 Blockly 편집기와 Python 편집기로 구성된다.
- 변환 계층은 블록에서 Python을 생성하는 정방향 생성기와, Python의 지원 구문을 블록 JSON 구조로 복원하는 역방향 변환기로 구성된다.
- 실행 계층은 WebAssembly 기반 Python 런타임, Worker 실행기, stage 렌더러, 입력/출력 브리지로 구성된다.
- 확장 계층은 라이브러리 패키지, 블록 정의, Python/JavaScript 생성기, AI 보조 생성기로 구성된다.
- 분석 계층은 사용자 이벤트 로그, 실행 로그, 오류 로그, 전환 로그, AI 사용 로그를 저장하고 분석한다.

### 5.3 양방향 변환 모델

Blockly는 본래 코드 생성 중심 구조를 제공한다 [R1]. 본 연구는 이를 확장하여 다음 두 경로를 제공한다.

1. `Blocks -> Python`
2. `Python -> Blocks`

정방향 경로는 Blockly의 Python generator를 기반으로 하되, 교육용 stage API와 라이브러리 패키지의 생성 규칙을 포함한다. 역방향 경로는 범용 Python 전체를 처리하는 파서가 아니라, 학습용 DSL과 교육용 API의 지원 구문을 대상으로 하는 들여쓰기 기반 구조 복원기다. 이 변환기는 토큰 수준 문법 분석 대신 다음 단계로 동작한다.

- 줄 단위 분할
- 들여쓰기 깊이에 따른 컨테이너 구조 생성
- 패턴 규칙에 따른 블록 타입 매핑
- 값 표현식의 블록 입력으로의 복원
- 형제 문장의 `next` 체인 복원
- 복원 불가능한 구문에 대한 주석 블록 또는 경고 노드 처리

이 방식의 장점은 완전한 Python 파서를 내장하지 않고도, 교육용 구문 집합 안에서는 높은 복원성과 직관적 편집 경험을 제공한다는 점이다. 단점은 지원되지 않는 Python 구문에 대해 부분적 손실이 발생할 수 있다는 점이며, 본 연구는 이를 `지원 구문 범위 명시`, `복원 실패 경고`, `원문 보존` 전략으로 완화한다.

### 5.4 브라우저 내부 Python 실행 모델

실행 경로는 기존의 `Blocks -> JavaScript -> browser execution`만으로는 실제 Python 생태계와의 연속성을 제공하기 어렵다. 따라서 본 연구는 최종 실행 의미를 Python에 맞추고, 브라우저 내부 Python 런타임으로 수렴하도록 설계한다.

- Blocks 모드에서는 workspace로부터 Python 코드를 생성한다.
- Python 모드에서는 사용자가 작성한 Python 코드를 그대로 사용한다.
- 두 경우 모두 최종 실행은 Worker 내부 Pyodide 런타임이 담당한다.

Pyodide는 Worker에서 비동기적으로 Python을 실행할 수 있으며, 메인 UI 스레드를 차단하지 않도록 권장된다 [R6]. 또한 `registerJsModule`을 통해 JavaScript 객체를 Python 모듈처럼 노출할 수 있다 [R7]. 본 시스템은 이 기능을 이용해 stage 제어 API, 출력 API, 시간 지연 API, 사용자 입력 API를 Python 코드에서 import 가능한 브리지 모듈로 제공한다.

### 5.5 이벤트 브리지와 상호작용 실행

교육용 프로그래밍 환경에서는 단순 출력뿐 아니라 다음 기능이 중요하다.

- 캐릭터 이동
- 회전
- 위치 변경
- 말풍선 출력
- 배경 전환
- 키보드/마우스 이벤트
- 시간 지연

브라우저 런타임은 DOM과 Canvas, 오디오, 입력 시스템에 직접 연결되지만 Python Worker는 별도 스레드에 있으므로 직접 DOM을 다룰 수 없다 [R6]. 따라서 본 시스템은 Python 코드가 직접 DOM을 다루지 않고, 브리지 모듈을 통해 명령을 전달하고 메인 스레드가 실제 렌더링을 수행하는 구조를 사용한다. 이 구조는 운영체제나 GUI 라이브러리에 대한 직접 의존 없이도 학습용 상호작용 과제를 지원한다.

### 5.6 패키지 및 라이브러리 확장 모델

Pyodide는 내장 패키지와 `micropip` 기반 패키지 설치를 지원하며, 현재 배포판에는 `opencv-python`을 포함한 다수 패키지가 포함된다 [R8]. 그러나 브라우저 환경에서 패키지가 import 가능하다는 사실과, 데스크톱 Python에서처럼 모든 기능이 동일하게 동작한다는 사실은 다르다. 예를 들어 카메라, 윈도우 출력, 운영체제 파일 시스템, 특정 네이티브 확장은 별도 제약이 있다. 따라서 본 연구는 라이브러리를 세 층으로 구분한다.

1. `code-generation only packages`
   Python 코드 생성만 지원하고, 브라우저 내부 실행은 보장하지 않는 패키지
2. `browser-executable packages`
   Pyodide에서 import와 실행이 가능한 패키지
3. `bridge-required packages`
   브라우저 자원 접근이 필요해 별도 JavaScript 브리지가 필요한 패키지

이 분류는 교육 현장에서 라이브러리의 기대 가능 범위를 명확히 하고, 과도한 오해를 방지한다.

### 5.7 AI 기반 라이브러리 보조 생성

AI 모듈은 사용자가 원하는 API 설명이나 샘플 사용법을 바탕으로 다음 산출물을 돕는다.

- 블록 정의 스켈레톤
- toolbox 카테고리 초안
- Python generator 초안
- JavaScript bridge 초안
- 교육용 예제 코드
- 블록 사용 설명

AI 출력은 곧바로 신뢰하는 것이 아니라, 정적 검증과 샌드박스 검증, 교사 승인 절차를 거쳐야 한다. 본 시스템은 AI를 `개발 보조자`와 `설명형 튜터`로 사용하며, 학습자가 과제를 통째로 위임하는 형태는 기본 정책상 제한한다.

### 5.8 학습 로그 및 분석 계층

학습 분석을 위해 다음 이벤트를 수집한다.

- 블록 추가/삭제/이동
- Python 편집 이벤트
- 블록-텍스트 전환 시점
- 실행 요청 횟수
- 실행 성공/실패
- 오류 유형
- AI 호출 횟수와 목적
- 과제 완료 시간
- 패키지 로드 이벤트

이 로그는 단순 이용통계가 아니라 전환 연구와 학습 효과 분석을 위한 핵심 데이터다. 예를 들어, Python 전환 이후 구문 오류가 급증하는 구간, AI 호출 직후 수정 폭이 커지는 구간, 특정 블록 유형에서 텍스트 전환이 잦은 구간을 추적할 수 있다.

## 6. 4장 연구문제와 평가 방법

### 6.1 연구문제

- RQ1. 제안 시스템은 blocks-only 및 text-only 환경 대비 전환 성과를 향상시키는가?
- RQ2. 제안 시스템의 브라우저 런타임은 교육용 상호작용 과제에 충분한 실행 충실도와 반응성을 제공하는가?
- RQ3. AI 보조 기능은 학습 효율을 높이되 과의존을 억제하는 방향으로 사용되는가?
- RQ4. 전환 로그와 오류 패턴은 학습 성과를 예측하는가?

### 6.2 시스템 수준 평가

시스템 수준 평가는 다음 항목으로 구성한다.

- 런타임 초기화 시간
- 패키지 로드 시간
- 첫 실행 지연
- 평균 이벤트 응답 시간
- 장시간 실행 안정성
- 로컬 CPython 대비 실행 결과 일치율
- 지원 API 집합에서의 실행 충실도

여기서 실행 충실도는 `동일 Python 프로그램에 대해 로컬 CPython과 브라우저 런타임이 동일하거나 허용 가능한 수준으로 동등한 결과를 산출하는 비율`로 정의한다.

### 6.3 교육 수준 평가

교육 수준 평가는 최소 4주 이상 수업 개입을 전제로 한다. 추천 설계는 `2 x 2 요인설계`다.

- 표현 환경: `text-only` vs `dual-modality`
- AI 지원: `off` vs `on`

종속변수는 다음과 같다.

- Python 지식검사
- 블록-텍스트 전환 과제 점수
- 디버깅 과제 점수
- 지연 사후검사 점수
- 자기효능감
- 인지부하
- 사용성
- 코드 품질
- AI 의존 지표

### 6.4 측정도구

- 프로그래밍 자기효능감: Ramalingam and Wiedenbeck의 프로그래밍 자기효능감 척도 계열 [R13]
- 인지부하: NASA-TLX [R14]
- 사용성: System Usability Scale
- 코드 품질: 가독성, 모듈화, 변수명, 오류 밀도, 요구사항 충족도를 포함한 루브릭
- 로그 지표: 실행 횟수, 오류 수, 전환 빈도, AI 호출 목적, 과제 완료 시간

### 6.5 통계 분석

- 사전점수 보정 사후비교: ANCOVA
- 반복측정 포함 분석: linear mixed-effects model
- 로그와 성과의 관계: 다중회귀 또는 generalized mixed model
- AI 사용 경로의 간접효과 분석: mediation analysis
- 코드 품질과 학습 성과의 관계: 상관 및 회귀

정량 결과는 다음 형식으로 제시한다.

- 평균과 표준편차
- 효과크기
- 95% 신뢰구간
- p-value
- 모델 계수와 적합도

## 7. 5장 예상 결과 서술 틀

아래 문단은 실제 결과가 나오면 숫자만 교체하도록 설계한 가안이다.

제안 시스템을 사용한 `dual-modality` 집단은 `text-only` 집단에 비해 사후 Python 전환 과제 점수가 유의하게 높았다 (`M = [ ]`, `SD = [ ]` vs `M = [ ]`, `SD = [ ]`, `F = [ ]`, `p = [ ]`, `partial eta squared = [ ]`). 특히 구문 오류 빈도는 `[ ]%` 감소하였고, 자료형 관련 오류는 `[ ]%` 감소하였다. 이는 블록과 텍스트 간 대응을 드러내는 양방향 표현 환경이 초보 학습자의 전환 비용을 낮춘다는 기존 논의를 확장한다.

또한 브라우저 내부 Python 런타임의 평균 초기화 시간은 `[ ]초`, 첫 실행 지연은 `[ ]ms`, 이벤트 응답 시간은 `[ ]ms`로 측정되었다. 로컬 CPython과의 결과 일치율은 지원 API 집합 기준 `[ ]%`였으며, 교육용 과제 범위에서 실용적 수준의 실행 충실도를 보였다.

AI 보조 기능은 과제 완료 시간을 `[ ]%` 단축시켰지만, 고성취 집단과 저성취 집단에서 사용 패턴이 달랐다. 고성취 집단은 AI를 주로 API 탐색과 오류 설명에 사용한 반면, 저성취 집단은 직접적 코드 생성 요청 비율이 높았다. 이는 AI 사용을 단순 이진 변수로 보지 않고, 목적 기반 사용 패턴으로 세분화해 해석해야 함을 시사한다.

## 8. 6장 결론 가안

본 연구는 블록 기반 입문 환경과 텍스트 기반 Python 환경 사이의 의미적 단절을 완화하기 위해, 양방향 블록-텍스트 변환과 브라우저 내 Python 인터프리터를 결합한 교육용 다중표현 프로그래밍 시스템을 제안하였다. 제안 시스템은 Blockly 기반 시각 편집기, Python 편집기, WebAssembly 기반 런타임, JavaScript 브리지, 동적 라이브러리 패키지, AI 보조 확장, 학습 로그 분석기로 구성되며, `표현의 다양성`과 `실행 의미의 일관성`을 동시에 달성하고자 하였다.

본 연구의 핵심 의의는 세 가지다. 첫째, 교육용 블록 환경을 범용 프로그래밍 언어 학습과 분리된 예비 단계로 보지 않고, 동일한 실행 의미 위의 다중표현 인터페이스로 재구성하였다. 둘째, 브라우저 내부 Python 실행을 통해 설치와 서버 의존성을 줄이면서도 실제 Python 생태계와의 연속성을 유지하였다. 셋째, 학습 성과를 정답률로만 평가하지 않고 전환 능력, 오류 유형, 행태 로그, 자기효능감, 인지부하를 통합해 분석할 수 있는 연구 프레임을 제시하였다.

향후 연구에서는 더 넓은 Python 구문 집합에 대한 역변환 지원, 브라우저 자원 접근이 필요한 패키지에 대한 브리지 자동생성, AI 사용의 책임 있는 통제 정책, 장기 추적 연구를 통한 유지 효과 검증이 필요하다.
