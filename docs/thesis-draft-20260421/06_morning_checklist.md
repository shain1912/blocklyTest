# 아침 체크리스트

## 1. 제일 먼저 볼 것

- 제목이 너무 서비스 소개처럼 보이지 않는가
- 서론이 너무 교육 일반론으로 흐르지 않는가
- 공헌이 `변환기`, `런타임`, `브리지`, `분석`으로 선명하게 보이는가
- 실험 설계가 자기보고만으로 끝나지 않는가

## 2. 반드시 오늘 중으로 채워야 하는 빈칸

- 시스템 이름 `[시스템명]`
- 실제 수업명
- 대상 학생 수
- 실험 가능 학기
- 교과 운영 방식
- AI 기능 범위

## 3. 지도교수님이 바로 물을 가능성이 높은 질문

### Q1. 이게 연구적으로 새로운 게 뭐냐

답 포인트:

- 블록 편집기 자체가 아님
- 양방향 변환 + 브라우저 Python 실행 + 의미 일관성 + 확장 패키지 + 로그 분석의 결합이 핵심

### Q2. 왜 백엔드 없이도 된다고 보느냐

답 포인트:

- 실행은 브라우저 내 Python 런타임
- 단, AI와 영구 저장은 얇은 백엔드가 더 적절
- 논문 핵심은 서버리스 자체가 아니라 설치 없는 실행과 학습 전환 지원

### Q3. 완전한 Python이 되느냐

답 포인트:

- 범용 Python 전체가 아니라 교육용 지원 구문과 지원 패키지 집합을 명확히 정의
- 실행 충실도는 로컬 CPython 비교 실험으로 보일 예정

### Q4. 왜 이게 박사 논문 수준이냐

답 포인트:

- 단순 UI 툴이 아니라 언어 변환, 런타임, 브리지, 패키지 모델, 학습 로그 분석이 결합된 시스템 연구
- 시스템 성능평가 + 교육 효과평가를 동시에 수행

## 4. 발표나 초안검사 때 약하게 보일 문장

- 혁신적인 에듀테크 플랫폼
- 미래형 학습 서비스
- 흥미와 몰입을 극대화
- AI 시대 교육혁신

## 5. 발표나 초안검사 때 강하게 보일 문장

- 의미 일관적 양방향 변환
- 브라우저 상주 Python 실행환경
- 이벤트 브리지 기반 상호작용 실행
- 패키지 확장과 실행 충실도
- 전환 능력에 대한 실증 분석

## 6. 실제 실험 전에 해야 할 준비

1. 지원 구문 목록 확정
2. 지원 라이브러리 목록 확정
3. 실패 시 fallback 정책 문서화
4. 로그 이벤트 스키마 확정
5. 사전/사후 문제지 초안 작성
6. 코드 품질 루브릭 작성
7. IRB용 설명문과 동의서 준비

## 7. 표와 그림부터 만들면 좋은 것

- 시스템 아키텍처 그림
- 변환 파이프라인 그림
- 실행 브리지 그림
- 기존 연구 비교표
- 지원 기능 비교표
- 실험 설계 표

## 8. 초안에서 꼭 지켜야 할 선

- 없는 결과를 사실처럼 쓰지 말 것
- 숫자는 모두 자리표시자로 둘 것
- 공개 자료에서 확인 가능한 것만 교수님 연구 경향으로 말할 것
- `교수님 취향 맞춤`은 문체와 프레이밍에 반영하되, 과도한 추정은 피할 것
